# WS1-WP1 Evaluation — SARA State Engine v1

**Evaluator:** Evaluator Agent
**Date:** 2026-05-31
**Scope:** WS1-WP1 only — bounded State Engine v1 slice
**Method:** Observable runtime behaviour only (no source/diff/build-status review for grading; the seed layer was read solely to craft a fault-injection input, and the verdict rests on observed behaviour)

---

## Overall recommendation: `iterate`

The WS1 slice is real, honest, and behaves well under fault. The WS0 placeholder
stub has genuinely been replaced by a contract-backed engine; seeded inputs are
labelled honestly at every layer; an invalid model is surfaced honestly as
`degraded` without crashing; and the briefing line is genuinely derived from and
consistent with the underlying model.

It falls short on **one** required criterion: criterion 2 names *current state,
current location, **and** confidence*. The running contract exposes a current
**state** consistently across backend and frontend, but exposes **no location
field and no confidence field anywhere** — neither in `/api/state`, `/api/health`,
nor the UI. Because the manager handoff that would define these terms is missing
(see below), I cannot confirm whether location/confidence are deferred-by-design
or an omission. Either way, as the criterion is literally written it is not
satisfied, so I hold the verdict at `iterate` rather than `converge`.

**This flips to `converge` immediately** if the manager confirms (in the missing
handoff) that location and confidence are explicitly out of WS1 scope, or if the
build adds a confidence signal (and location, if intended) exposed consistently
on both surfaces. Nothing here is broken, dishonest, or regressed — hence
`iterate`, not `block`.

## Required criteria

| # | Criterion | Status |
|---|-----------|--------|
| 1 | WS0 stub replaced by a real runtime model | **PASS** |
| 2 | Current state, current location, and confidence all exposed consistently | **FAIL** (state ✓ consistent; location ✗ absent; confidence ✗ absent) |

| 5 | Invalid / failed model generation surfaced honestly | **PASS** |

### 1. WS0 stub replaced by a real runtime model — PASS
- WS0 served `placeholder:true`, `schemaVersion:0`, `domains:{}`.
- WS1 `/api/state` now serves `schemaVersion:1`, `contract:"state-engine-v1"`,
  `dataSource:"seed"`, and four populated domains (`queue`, `focus`, `people`,
  `vault`) assembled into one shared model, plus a derived `briefing` and a
  `meta` validation block. No `placeholder` flag remains.
- `/api/health` confirms the shift: `runtime:"WS1-WP1"`,
  `contract:"state-engine-v1"`, `schemaVersion:1`, `valid:true`. The model is
  real and self-validating, not a stub.

### 2. Current state / location / confidence exposed consistently — FAIL
- **Current state — exposed and consistent.** The assembled model
  (`sara.status:"online"`, `meta.valid`, per-domain `status` values, briefing) is
  served by the backend and the headline values (`runtime`, `contract`, `valid`,
  `servedAt`, briefing, domain summaries) are rendered identically in the
  frontend. No field-presence drift observed between the two surfaces for what is
  exposed.
- **Current location — absent.** No `location` field exists anywhere in
  `/api/state`, `/api/health`, or the rendered UI (verified by full-payload key
  inspection: `"location" present: false`).
- **Confidence — absent.** No `confidence` field exists anywhere
  (`"confidence" present: false`). The briefing carries `derivedFrom`
  (provenance) but no confidence signal.
- Because two of the three named values are not exposed at all, the criterion's
  conjunction ("all exposed consistently") is not met. This is the sole basis for
  the `iterate` verdict.

### 3. Frontend consumes and displays the values — PASS
- Rendered the production single-process runtime (backend serving the built
  `frontend/dist`) in a real browser.
- The UI reads `/api/state` and displays: a **CONNECTED** badge (earned only
  after a successful read — confirmed by the degraded run still connecting),
  the derived briefing line, `runtime / contract (v1) / valid / served at`
  metadata, and a per-domain summary list (QUEUE / FOCUS / PEOPLE / VAULT).
- Same-origin only: a cross-port `fetch` from the page was correctly blocked by
  CORS — no loose cross-origin data path.

### 4. Seeded inputs labelled honestly — PASS
- Root `dataSource:"seed"`; every domain stamped `source:"seed"`.
- `sara.note`: *"…Inputs are seeded (hardcoded), not yet wired to real sources
  (WS1 scope)."*
- UI banner: *"Seed data — State Engine v1 contract is live, inputs are hardcoded
  (WS1 scope)."* plus a matching footer. The slice never overstates the data as
  live.

### 5. Invalid / failed model generation surfaced honestly — PASS
Fault-injected an invalid `queue` domain via the swappable seed layer (the
documented integration seam), ran a separate backend instance, observed, then
reverted via `git checkout`:
- `/api/health` flipped to `status:"degraded"`, `valid:false`.
- `/api/state` returned `meta.valid:false` with a **precise error list**
  (`domains.queue.open is missing`, `…breaching is missing`, `…sections is
  missing`, `…source must be a string`, `…summary must be a string`).
- The engine **did not crash** — it served the invalid model transparently and
  the briefing **gracefully dropped** the broken queue clause (line fell back to
  Nathan + focus only).
- The frontend reflected this honestly: `valid: false` shown in metadata, and the
  QUEUE tile rendered with **no summary** rather than fabricating data.

## Holdout checks (mature-state-engine risks)

| Holdout | Result |
|---------|--------|
| Inconsistent field presence between backend and frontend views | **PASS (partial)** — see note |
| Partially missing domain data | **PASS** |
| Stale or contradictory confidence/state values | **PASS (state); N/A (confidence absent)** |
| Invalid model handling + honest health surfacing | **PASS** |

- **Field presence:** for everything the contract *does* expose, backend and
  frontend agree (no drift). The caveat is that the frontend surfaces domain
  *summaries* only, not full domain detail — acceptable for a proof surface, but
  it means the richer state (ticket lists, member roster, vault picks) is not yet
  consumer-visible. The genuine gap is the absence of location/confidence on
  *both* surfaces (criterion 2), not a backend↔frontend mismatch.
- **Partially missing domain data:** under the injected fault, one domain went
  invalid while the other three stayed valid; the system degraded that one domain
  (blank tile, dropped briefing clause) without taking down the rest. Honest
  partial degradation.
- **Stale / contradictory values:** in the clean run the briefing is genuinely
  derived and internally consistent — `queue.breaching:2` ↔ "2 tickets are
  breaching SLA"; `people` Nathan `status:"slipping"` / "no response since
  Wednesday" ↔ briefing; `focus.current.title` ↔ "Start with: Prep Willem's
  probation review"; `derivedFrom:["queue","people","focus"]` matches the three
  clauses. No stale or contradictory values found. (Confidence could not be
  checked for staleness — it does not exist.)
- **Invalid model + health:** see criterion 5 — strongest area of the slice.

## Scoped regressions / nits (non-blocking)

1. **`derivedFrom` over-claims under partial failure.** In the faulted run the
   briefing line correctly dropped the queue clause, but `derivedFrom` still
   listed `"queue"`. Provenance should track what actually contributed. Cosmetic;
   no functional impact.
2. **Degraded state is under-signalled in the UI.** When `valid:false`, the only
   UI signals are the small `valid: false` metadata field and an empty QUEUE
   tile; the header still reads **CONNECTED** and the banner still reads "contract
   is live". It is honest (nothing fabricated) but not prominent — an operator
   could miss it. A visible degraded indicator would be a cheap improvement.
3. **Frontend shows summaries only.** Rich domain detail present in `/api/state`
   is not surfaced in the UI. Fine for a connectivity/proof surface; noted for
   whichever WP owns the real SARA view.

No correctness regressions found.

## Sufficiency for WS2 planning

**Sufficient with one caveat to close first.** The engine, the enforced
`state-engine-v1` contract, the seam between the engine and its swappable input
providers, and the honest health/validation path are all real and working — a
clean foundation for WS2 to land live data providers against without touching the
engine or contract. The one thing WS2 planning should *not* inherit unresolved is
criterion 2: the programme must decide explicitly whether **location** and
**confidence** belong in the state contract and, if so, get them exposed
consistently on both surfaces before they become harder to retrofit. That
decision is small but should be made now, not discovered later.

**Recommended before/with WS2:**
- Produce the missing manager handoff (`manager_log/`) — its absence directly
  weakened the criterion-2 judgement here and was already flagged at WS0.
- Resolve location/confidence: either add them to the contract + UI, or record
  the explicit deferral that turns criterion 2 into a PASS.
