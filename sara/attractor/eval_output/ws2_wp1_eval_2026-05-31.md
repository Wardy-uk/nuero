2026-05-31

## Evaluator
Evaluator Agent (runtime-only standard)

## Method
Judged **only observable runtime behaviour** in the governed `nuero` workspace.
Source code and diffs were **not** read to decide correctness. Evidence was gathered
by building and running the system and observing the live page:

- Rebuilt the frontend (`npm run build`, Vite 5.4.21) — confirmed it emits a clean `dist/`.
- Started the backend via the preview launcher (`sara` config, port 3005) so the
  preview tooling could attach to the live page.
- Probed `/api/health` and `/api/state` over HTTP.
- Read the running page through the accessibility snapshot + DOM/computed-style
  assertions (`preview_eval`), exercised the view switcher by clicking, and tested
  two touch viewports.
- Launcher (criterion 1) is Pi-targeted and cannot be executed on this Windows dev
  machine, so it was assessed against the criterion as written — *does an installable
  launcher / followable install instructions exist* — by inspecting the delivered
  launcher artefacts as deliverables (not as source-for-screen-correctness).

## Runtime-only evidence

### Build & serving
- `npm run build` → **39 modules transformed**, `dist/` emitted clean (~0.6s), no errors.
- Backend booted on port 3005 and served the build.
- `GET /api/health` → `{"status":"ok","sara":"online","contract":"state-engine-v1","schemaVersion":1,"dataSource":"seed","valid":true,"location":"Office — Wilmslow","confidence":{"level":"moderate","score":0.6}}`
- `GET /api/state` → contract valid, `dataSource:"seed"`, focus goal "Prep Willem's probation review".

### Criterion 1 — Launcher / install instructions exist (PASS)
- `desktop/SARA.desktop` is a valid XDG entry: `Exec=/mnt/data/nuero/sara/scripts/start-sara.sh`,
  `Icon=/mnt/data/nuero/sara/desktop/sara.svg` (icon file present), `Terminal=false`.
- `scripts/start-sara.sh` and `desktop/sara.svg` exist; `desktop/README.md` gives concrete,
  followable install steps (`chmod +x`, copy to `~/Desktop` and `~/.local/share/applications`,
  `update-desktop-database`), a terminal fallback (`bash scripts/start-sara.sh`), and a
  `SARA_URL` override for a different host/port.
- The Exec path matches the documented WS0 deploy path (`/mnt/data/nuero/sara`) and the README
  explicitly documents editing it if SARA lives elsewhere — internally consistent.
- *Holdout — "launcher exists on disk but cannot be followed in practice":* **not triggered.**
  The instructions are concrete, self-consistent, and include a path override and a no-icon
  terminal fallback. (Execution itself was not performed — it is Pi/Chromium-kiosk-targeted and
  out of scope on the Windows dev host.)

### Criterion 2 — Mission Control renders (PASS)
- Accessibility snapshot of the live page shows all nine required elements present and populated
  from shared state: SARA header + `ONLINE` state + `SEED DATA` pill; live clock (`10:05` →
  later `10:06`) / `Sunday 31 May`; `LOCATION` = "Office — Wilmslow"; `CONFIDENCE` =
  "Moderate · 0.6"; `CURRENT GOAL` = "Prep Willem's probation review" + reason; What Matters Now
  (3 items); Up Next (3 items); Quick Actions (4 buttons).
- **No console logs at any level** (no errors, no warnings) across the whole session.

### Criterion 3 — Placeholder / shared-state data appears (PASS)
- The honest `SEED DATA` pill renders (driven by `/api/state` `dataSource:"seed"`).
- Placeholder presentation content renders on screen: *What Matters Now* (3), *Up Next* (3,
  time-stamped 11:00 / 13:30 / 15:00), *Quick Actions* (4).
- Authoritative seed values rendered on screen match `/api/state` **exactly** — location,
  confidence level+score, and current goal title/reason are identical to the API payload,
  confirming the screen renders shared state rather than a divergent local copy.

### Criterion 4 — Quick action buttons render (PASS)
- Four buttons render: **Capture, Open Queue, Start Focus, Daily Brief**.
- At 375px they form a 2-up grid with 84px-tall tap targets (well above the 44px touch minimum).

### Criterion 5 — Architecture supports future screens (PASS)
- Six views are declared in the switcher: `Mission Control` (available) and
  `Executive Dashboard`, `Presence`, `Focus`, `Companion`, `Stream Deck` (each marked `SOON`).
- **Verified at runtime by clicking:** selecting `Stream Deck` set its chip active
  (`aria-pressed="true"`, `--active` class) and the router replaced `main` with the
  `PlannedView` ("Planned view / Stream Deck / Large touch-action grid… reserved by SARA's
  many-views architecture"). Clicking `Mission Control` restored it (`main` region label back
  to "Mission Control", quick actions present). Round-trip confirmed.
- *Holdout — "future-view scaffolding present in name only, not wired through a current-view
  concept":* **not triggered.** The planned views are genuinely routed through the
  current-view selection — selecting one unmounts Mission Control and mounts a real placeholder
  screen, proving the app is not hardcoded to one home screen.

### Criterion 6 — No screen owns authoritative state (PASS)
- The clock is a **shared, provider-driven live clock**: it advanced `10:05` → `10:06` during
  the session with no reload, and stayed consistent across view switches — i.e. it is not a
  screen-local timer that resets on mount.
- On-screen location / confidence / goal match `/api/state` exactly; switching to a planned view
  and back preserves identical values. No observable divergence that would indicate a
  screen-held copy of authoritative state.
- *Holdout — "state duplicated inside screen components":* **not triggered** at runtime. All
  authoritative fields are single-sourced from the engine model; even time comes from the shared
  layer, not the screen.

## Holdout findings

- **No adverse holdout findings.** All four hidden checks (state duplicated in screens; launcher
  that can't be followed; future-view scaffolding in name only; responsive/touch regressions on
  the target form factor) were probed and none surfaced a defect.
- **Evaluator-method note (not a build defect):** a synchronous DOM read taken in the *same tick*
  as a switcher click briefly showed stale Mission Control markup — React had not yet flushed.
  On the next tick the PlannedView was correctly mounted. This was a test-harness timing artifact,
  not an app behaviour; flagged only so it isn't mistaken for a render bug.

## Scoped regressions / non-blocking nits

None of the following block convergence:

1. **Switcher chips are 40px tall** on the 800×480 landscape kiosk form factor — just under the
   44px touch-target ideal. The primary quick-action buttons are fine (84px). Secondary controls,
   minor.
2. **Vertical scroll on a short kiosk display:** at 800×480 (official Pi 7" touchscreen),
   Mission Control content exceeds the viewport height (~337px of vertical scroll). Acceptable on
   a touchscreen but worth noting for the target form factor; no horizontal overflow at 375px,
   800×480, or desktop.
3. **`preview_screenshot` times out (~30s)** because the 1s shared-clock interval keeps the
   renderer from reaching idle. This is a tooling limitation, not an app defect — DOM /
   accessibility / computed-style evidence was used instead, and the console is clean. No visual
   screenshot was captured.

---

## Is WS2 sufficient to unlock later view work?

**Yes.** The view-system foundation is wired through a real current-view concept and verified at
runtime, not present in name only:

- A view registry + router + manual switcher + a `PlannedView` fallback are all live, and
  selecting any of the five planned views genuinely mounts a placeholder screen.
- A single shared read-only state layer (authoritative engine state from `/api/state` + a
  swappable placeholder-presentation seam + a shared clock) is the one source every screen reads.

Later workstreams can add a screen by registering a view and mounting a component against the
existing shared-state hook, without re-architecting routing or state ownership. The non-blocking
nits above (touch-target height, kiosk vertical scroll) are layout polish for whoever owns the
final kiosk form factor and do not gate later view work.
