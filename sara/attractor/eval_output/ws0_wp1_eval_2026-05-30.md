### 1. Startup path — PASS (platform caveat)
Intended production model is a single process: the backend serves the built
frontend, registered with PM2 under systemd (`pm2 save`) so it auto-starts on
Pi reboot with no manual launch.

- Ran the equivalent production process (`node server.js`, the same script PM2
  invokes). It boots cleanly and logs `[SARA WS0-WP1] backend listening on
  http://0.0.0.0:3005`.
- Binds `0.0.0.0` → reachable over Tailscale, matching the documented access URL.
- PM2 definition is correct for boot persistence: `autorestart: true`,
  `max_restarts: 10`, fixed `script`/`cwd`, `SARA_PORT` env.

**Caveat:** the *auto-start-on-boot* trigger (systemd → PM2 resurrect) could not
be exercised on this Windows host — no Pi/systemd available. The substance of the
claim (clean bring-up + survives a stop/start cycle) was runtime-verified; only
the systemd boot hook is asserted from standard, correct config rather than
executed. Not a blocker.

### 2. Frontend reachable (always-on mode) — PASS
- `GET /` → `200`, serves the built SPA `index.html` (title "SARA").
- Built assets serve with correct MIME types: JS `200 application/javascript`,
  CSS `200 text/css`.
- Rendered in a real browser: the React app mounts and paints the runtime UI.
- SPA fallback works: an unknown deep route (`/some/deep/route`) returns
  `200 text/html` (serves the app shell), as intended for a PWA.

### 3. Backend reachable on expected runtime path — PASS
- `GET /api/health` → `200`
  `{"status":"ok","sara":"online","runtime":"WS0-WP1","placeholder":true,...}`
- `GET /api/state` → `200` with `schemaVersion`, `placeholder:true`, `domains:{}`,
  and an honest `sara.note`.
- API namespace is cleanly separated from the SPA fallback: unknown `/api/*`
  (`/api/does-not-exist`) correctly returns `404` rather than leaking the SPA
  HTML. Good hygiene.

### 4. Frontend/backend communication actually works — PASS
- In the running browser the app issued `GET /api/state → 200` and rendered the
  returned values directly into the DOM (`runtime: WS0-WP1`, `sara.status:
  online`, live `backend started` / `served at` timestamps).
- The connectivity indicator turned **green / "CONNECTED"** only after a
  successful state read — i.e. the green state is earned from a real round-trip,
  not hardcoded.

### 5. Honest failure surfacing — PASS
- **Placeholder honesty:** every response and the UI itself are explicitly
  flagged as a placeholder ("Placeholder state — central State Engine not wired
  yet (WS0 scope)", `placeholder:true`). The slice does not overstate what it is.
- **Down state:** with the backend stopped, both `GET /` and `GET /api/health`
  return connection-refused (`000`) — nothing stale is served, no false
  positive. When the runtime is down, it is visibly down.
- **Failure UI exists:** the shipped client implements a full connection state
  machine — `connecting → connected / disconnected / error` — with `catch`
  handling around the fetch and "unreachable" operator copy. The UI is built to
  report a bad backend link, not freeze on a fake green.

## Holdout checks (mature-runtime risks)

| Holdout | Result |
|---------|--------|
| Restart persistence after stop/start cycle | **PASS** |
| Honest degradation when backend communication fails | **PASS** |
| Empty / placeholder runtime payload handling | **PASS** |
| Configuration dependency failure surfacing | **PASS** |

- **Restart persistence:** boot → kill → reboot cycle. Process came back up
  cleanly each time and served `200` again; `startedAt` advanced on each boot,
  confirming the in-memory state resets on restart — exactly the documented WS0
  behaviour, surfaced honestly via `placeholder:true`. (PM2 `autorestart` would
  perform this recovery automatically on the Pi.)
- **Honest degradation:** the app fetches state once on mount (no polling — 0
  calls observed over a 16s window with fetch forced to fail). In the
  single-process production model a down backend means the page itself is
  unreachable (verified: connection refused), so an operator cannot land on a
  lying UI. The client additionally ships explicit `disconnected`/`error` states
  for the dev/proxy case where the page loads but the API is unreachable.
- **Empty payload handling:** `/api/state` returns `domains:{}`; the UI rendered
  this without error, showing the placeholder banner instead of crashing or
  rendering blank.
- **Config dependency failure:** `SARA_PORT` override verified live (served on
  `3099`, nothing on `3005`). A port conflict fails **loud and fast** —
  `EADDRINUSE` crashes the process with a clear error and non-zero exit rather
  than silently half-starting. Loud failure is the honest outcome here; under
  PM2 the crash/restart-cap would be operator-visible.
