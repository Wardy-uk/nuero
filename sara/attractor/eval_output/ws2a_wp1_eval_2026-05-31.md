# WS2A-WP1 Evaluation — Additional Views v0 (Executive Dashboard + Presence Mode)

## Date

2026-05-31

## Work Package

`WS2A-WP1`

## Method

Judged **observable runtime behaviour only** in the governed `nuero` workspace. No
source code or diffs were read to decide correctness. The frontend was built fresh
(`vite build`, 43 modules, clean) so `dist/` reflected current source, then the SARA
backend served that build on port 3005 (the `sara` launch config). All evidence below
comes from the running page: accessibility snapshots, live DOM/computed-state
assertions, the `/api` responses, and the browser console.

A note on tooling, not the app: the `preview_click` synthetic click did not trigger the
React onClick handler, so view switches were driven via native element `.click()` in the
running page. Native clicks and the app's own React handlers fire correctly — this is a
harness artifact, **not** an application defect (the same chip, when clicked natively,
switches the view every time). `preview_screenshot` was not relied on (the 1-second
shared clock interval keeps the renderer from idling); structural snapshots + DOM
assertions are the stronger evidence here.

## Overall Recommendation

All five observable success criteria pass. No failure conditions triggered. No scoped

`schemaVersion: 1`, `dataSource: seed`, `valid: true`, `confidence: moderate · 0.6`,
`location: Office — Wilmslow`, `runtime: WS1-WP1`. The WS1 state contract is unchanged
and is the single payload every view reads.

### 2. Presence Mode renders — PASS

Selecting the Presence chip mounts `.presence` (aria-label "Presence"). DOM text of the
running view: large clock `10:51`, date `Sunday 31 May`, location `Office — Wilmslow`,
`SARA online`, the same engine briefing line verbatim, `Now: Prep Willem's probation
review`, `Next · 11:00: Stand-up with the support team`. Calm/ambient layout as
specified — a strict subtraction of the same state, no operational detail.

### 3. Both selectable through the current-view system — PASS

The switcher (`nav "SARA views"`) lists all six views. Executive Dashboard and Presence
appear **without** the `SOON` tag; Focus / Companion / Stream Deck retain it. Verified
the full round-trip at runtime by active-chip + mounted-root assertions:

- Mission Control (default) → `.mc` mounted.
- → Executive Dashboard: `.ed` mounted, active chip "Executive Dashboard", `.mc` gone.
- → Presence: `.presence` mounted, active chip "Presence", `.ed` gone.
- → Mission Control: `.mc` restored, active chip "Mission Control", `.presence` gone.

Only one view root is mounted at a time; the app is not locked to a single home screen.

### 4. Both read shared state, owning no authoritative state — PASS

Every value shown by both new views matches the single `/api/state` payload and is
identical to what Mission Control shows from the same source:

- Breaching SLA = **2**, Open = **4**, Confidence = **moderate · 0.6**, location =
  `Office — Wilmslow`, seed pill present on all three views.
- The engine-derived briefing string is rendered **byte-for-byte identical** in Executive
  Dashboard and Presence — neither re-derives or rewords it.
- Presence "Now" = focus current title; "Next · 11:00" = first shared Up Next placeholder
  item — the same shared placeholder layer Mission Control reads, not screen-local data.

No second source of truth, no per-screen state fork observed at runtime; the views differ
only in formatting/density, not in data.

### 5. Mission Control still works — PASS

Mission Control loads as the default view and, after switching away to both new views and
back, is restored intact: header (SARA / online / seed data / clock / date), Location,
Confidence, Current goal, What Matters Now, Up Next, and Quick Actions all present and
populated from shared state. No change in behaviour from the presentation-layer
generalisation.

## Scoped Regressions or Blockers

None observed at runtime.

## Is WS2A sufficient to unlock later screen work?

Yes. The many-views architecture is now demonstrated with **three** real,
shared-state-only views plus three planned-view placeholders all rendering and switching
cleanly through one ViewRouter/ViewSwitcher path and one shared presentation layer.
Adding the remaining planned screens (Focus / Companion / Stream Deck) is reduced to
flipping a registry status, adding a router case, and adding a screen component that
consumes `useSaraState()` — with no change to the WS1 state contract. The pattern holds
under runtime switching, so later screen work is unblocked.
