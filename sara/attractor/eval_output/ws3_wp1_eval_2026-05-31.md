**`converge`**

All four observable success criteria pass. The telemetry bridge ingests Home

## Method

Judged entirely from observable runtime behaviour. The SARA backend was run in
three telemetry states and probed over HTTP; the existing frontend was run against
it and a screen was loaded in a browser. To exercise the live-ingestion path
without production HA credentials, a local mock Home Assistant REST server was
stood up serving the three configured entities
(`person.nick=home`, `binary_sensor.office_occupancy=on`,
`sensor.office_temperature=21.4°C`) on `GET /api/states/<entity_id>`, matching HA's
real response shape. SARA was pointed at it with a 1s poll. Failure was induced by
killing the mock mid-run and re-probing.

## Runtime Evidence

### Criterion 1 — telemetry ingested (PASS)

Backend pointed at live (mock) HA, three bounded slots configured. Boot log:
`[SARA HA] telemetry polling every 1000ms against http://127.0.0.1:8199`.

`GET /api/telemetry`:
```
{"source":"home-assistant","available":true,"reason":null,
 "polledAt":"2026-05-31T13:04:50.964Z",
 "signals":{
   "location":{"entityId":"person.nick","state":"home","zone":"home","label":"Home"},
   "presence":{"entityId":"binary_sensor.office_occupancy","state":"on","present":true,"label":"Office Occupancy"},
   "environment":{"entityId":"sensor.office_temperature","state":"21.4","unit":"°C","label":"Office Temperature: 21.4°C"}}}
```
All three bounded signals were read from HA over REST and normalised. Nothing
auto-discovered beyond the three configured slots.

### Criterion 2 — shared state model reflects telemetry (PASS)

Same live state, `GET /api/state` (excerpt):
```
"location":{"source":"home-assistant","label":"Home","context":"home",
            "since":"2026-05-31T13:04:50.964Z","entityId":"person.nick"},
"telemetry":{"source":"home-assistant","available":true,"ageMs":959,
             "signals":{ ...all three populated... }},
"meta":{"valid":true,"errors":[],"domainCount":4}
```
`location` flipped to the HA source (`label:"Home"`) where the seed value is
`"Office — Wilmslow"`; the `telemetry` block carried the live signals with a
computed `ageMs`; the model stayed contract-valid (`meta.valid:true`,
`schemaVersion:1`, `contract:"state-engine-v1"`). The same `buildModel()` output
fed `/api/health`, which agreed: `locationSource:"home-assistant"`,
`telemetry.available:true`.

End-to-end into a screen: the Mission Control view, loaded in the browser against
this backend, rendered `LOCATION: Home` — the live HA value, not the seed —
confirming telemetry reaches the rendered UI through the shared state, not just the
API.

### Criterion 3 — absence/failure surfaced honestly (PASS)

Two unavailable states observed, each with a distinct machine-readable reason:

- **Not configured** (no HA env): boot log
  `telemetry bridge idle — not configured. Screens use fallback (location from seed)`.
  `/api/telemetry` → `{"available":false,"reason":"not-configured","signals":{location:null,presence:null,environment:null}}`.
  `/api/health` → `telemetry.available:false`, `reason:"not-configured"`, `locationSource:"seed"`.
  `/api/state` → `location.source:"seed"`, `location.telemetry:"fallback"`,
  `telemetry.available:false`, `meta.valid:true`.

- **Unreachable** (HA killed mid-run): after the next poll cycle, `/api/telemetry`
  → `{"available":false,"reason":"unreachable","detail":"no configured HA entity could be read from http://127.0.0.1:8199","signals":{all null}}`.
  `/api/health` and `/api/state` both reverted to `locationSource:"seed"` /
  `location.source:"seed"` (`"Office — Wilmslow"`), `telemetry.available:false`,
  `reason:"unreachable"`, `meta.valid:true`.

In every state `/api/health` and `/api/state` agreed about live-vs-unavailable —
no split-brain. The engine recovered from live → unreachable without throwing.

### Criterion 4 — existing screens remain functional (PASS)

- Frontend build (`vite build`): 49 modules transformed, `dist/` emitted clean in
  ~0.6s, no errors. Frontend unchanged from WS2A.
- Mission Control loaded in-browser against the WS3 backend with **live HA** —
  rendered fully (goal, "what matters now", schedule, `LOCATION: Home`,
  `CONFIDENCE Moderate · 0.6`). No console errors (error-level log empty).
- HA then killed; screen reloaded against the now-unreachable backend — **still
  rendered fully** (no offline/error state shown), with `LOCATION` reverting to the
  seed value `"Office — Wilmslow"`. Screens consume the same contract-valid
  `/api/state` payload they did at WS2A regardless of HA state.

## Scoped Regressions or Blockers

None observed. No screen broke when HA was absent or failed; no health/state
disagreement; the slice did not expand into new screens or automatic view selection
(the ViewSwitcher and six existing views are unchanged).

## Scope / Limitations Noted (not blockers)

- Live ingestion was exercised against a local mock HA, not the production HA
  instance (no production credentials in this environment). The full path —
  config → poll → normalise → cache → fold into `buildModel()` → screen — was
  exercised end-to-end against the mock, which mirrors HA's REST contract.
- Domain inputs (queue/people/focus/vault) remain seeded by design for this slice;
  honestly flagged via root `dataSource:"seed"`. Location and telemetry carry their
  own live source when HA is up. This is within WS3-WP1 scope (telemetry slice, not
  domain integration).

## Note on Unlocking Later Context Work

WS3 is **sufficient** to unlock later context work. The provider/State-Engine seam
behaves correctly under runtime conditions: telemetry feeds the one shared model
without blocking or destabilising assembly, the model stays contract-valid whether
HA is live, absent, or failing, and `/api/health` / `/api/state` never disagree.
That gives later workstreams a trustworthy, honestly-sourced shared state to build
context on. Two things to carry forward (not WS3 gaps): only `location` currently
consumes a live signal — presence/environment are surfaced but not yet folded into
any derived domain or decision — and live ingestion still needs one confirmation run
against the real production HA before later work depends on production telemetry.
